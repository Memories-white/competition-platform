# 基于Docker的云计算竞赛环境自动构建系统 — 系统架构文档

## 1. 系统概述

本系统实现了一个基于 Docker 的云计算竞赛平台，核心功能是**自动构建、部署和管理**竞赛所需的隔离容器环境。系统采用 B/S 架构，管理员通过 Web 界面创建竞赛、配置题目、触发自动构建；选手通过浏览器直接访问自己的容器环境（URL 直连），或通过 SSH / docker exec 命令行操作，提交判题后系统自动评分。

系统设计支持**单机本地部署**和**云服务器远程部署**两种模式，通过 Docker SDK 的 `DOCKER_HOST` 环境变量即可切换，无需修改代码。

作为扩展功能，系统还支持**试卷考试**模式（单选题、判断题、填空题），管理员可在线组卷，选手在线作答并自动评分，与 Docker 实操环境并行使用。

### 技术栈

| 层级 | 技术 |
|------|------|
| Web 框架 | Flask (Python) |
| 数据库 | SQLite (开发) / 可扩展至 PostgreSQL |
| ORM | SQLAlchemy |
| 容器引擎 | Docker (docker-py SDK) |
| 实时通信 | Flask-SocketIO (WebSocket) |
| 任务调度 | APScheduler |
| 前端 | Jinja2 + Bootstrap 5 + Vanilla JS |

## 3. 云部署架构

### 3.1 双模式部署设计

系统通过 Docker SDK 抽象底层容器运行时，天然支持两种部署模式：

```
┌─ 模式 A: 单机本地部署 ─────────────────────┐
│                                            │
│  Flask App ── docker-py ── Docker Daemon   │
│  (localhost)              (localhost)       │
│                                            │
│  适用场景: 开发测试、小规模竞赛               │
│  DOCKER_HOST = (空)  → docker.from_env()   │
└────────────────────────────────────────────┘

┌─ 模式 B: 云服务器部署 ───────────────────────────────┐
│                                                      │
│  选手浏览器 ──HTTP──▶ Flask App (云服务器 :5000)       │
│                           │                           │
│                           ▼                           │
│                      docker-py                        │
│                           │                           │
│                           ▼                           │
│               Docker Daemon (云服务器)                  │
│                    │                                  │
│        ┌───────────┼───────────┐                     │
│        ▼           ▼           ▼                     │
│    [容器1]     [容器2]     [容器N]                     │
│    :30000     :30001     :3000N                       │
│                                                      │
│  选手访问容器: http://<云服务器公网IP>:30000            │
│                                                      │
│  配置方式:                                            │
│    export HOST_IP=<云服务器公网IP>                     │
│    export DOCKER_HOST=(空，使用本地 Docker)            │
│                                                      │
│  适用场景: 正式竞赛、大规模并发                          │
└──────────────────────────────────────────────────────┘
```

### 3.2 云部署配置清单

| 配置项 | 环境变量 | 默认值 | 说明 |
|--------|---------|--------|------|
| 主机 IP | `HOST_IP` | `localhost` | 选手访问容器的公网 IP 或域名 |
| Docker 主机 | `DOCKER_HOST` | (空=本地) | 远程 Docker 引擎地址，如 `tcp://10.0.0.5:2375` |
| Secret Key | `SECRET_KEY` | 内置默认值 | 生产环境必须修改 |

### 3.3 容器 URL 直连

每个选手容器部署后，系统自动生成访问 URL：

```
http://{HOST_IP}:{host_port}
```

选手在环境页面可直接点击打开容器内的 Web 服务（如 Nginx、Flask、Express），也可一键复制链接。URL 中的 IP 地址通过 `HOST_IP` 配置，部署到云服务器时自动变为公网可访问地址。

### 3.4 容器生命周期管理

容器生命周期覆盖**创建 → 运行 → 判题 → 销毁**四个阶段：

```
创建 (deploy)  →  运行 (running)  →  判题 (judging)  →  销毁 (destroyed)
      │                │                   │                  │
      │          [选手操作容器]       [30s 自动判题]      [手动/自动]
      │                │                   │                  │
      ▼                ▼                   ▼                  ▼
  ┌─────────┐   ┌──────────┐      ┌──────────┐      ┌──────────────┐
  │ 分配端口  │   │ URL 直连  │      │ port 检测 │      │ stop 容器     │
  │ 资源限制  │   │ SSH 登录  │      │ command   │      │ remove 容器   │
  │ 网络隔离  │   │ docker exec│     │ file 检测 │      │ 标记 removed  │
  └─────────┘   └──────────┘      └──────────┘      └──────────────┘
```

**手动销毁：** 选手在环境页面点击「销毁环境」按钮，确认后立即停止并删除容器，环境状态标记为 `removed`。

**自动清理（竞赛结束）：** 调度器每 120 秒扫描状态为 `finished` 的竞赛，执行最后一轮判题后，停止并删除该竞赛下所有容器，回收端口和计算资源。

**销毁安全机制：**
- 销毁前弹出确认对话框，防止误操作
- 按钮在销毁过程中禁用，防止重复提交
- 容器删除后端口自动释放，可被新环境复用

## 4. 核心模块设计

### 4.1 镜像构建引擎 (builder.py)

```
输入: Dockerfile 文本 / 模板名称
  │
  ▼
┌─────────────────────┐
│  Dockerfile 解析     │ ← 自动提取 EXPOSE 端口
│  (get_expose_port)   │
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│  Docker SDK Build    │ ← 构建容器镜像
│  (docker.images.build)│
└────────┬────────────┘
         │
    ┌────┴────┐
    ▼         ▼
  成功       失败
    │         │
    ▼         ▼
  返回 tag   重试 (最多2次, 间隔3s)
             │
         ┌───┴───┐
         ▼       ▼
       成功     失败 → 返回错误
```

**关键特性：**
- 支持从 Dockerfile 文本直接构建
- 支持从预置模板目录构建
- **构建失败自动重试**（最多 2 次，间隔 3 秒）
- 带镜像加速下载提示的错误格式化
- 连接超时控制（5 秒）

### 4.2 容器管理引擎 (manager.py)

```
┌────────────────────────────────────────────┐
│            容器生命周期管理                    │
│                                            │
│  create_container()                        │
│    ├─ 申请空闲端口 (30000-40000)             │
│    ├─ 清理同名旧容器                         │
│    ├─ 设置资源限制 (CPU/Memory)             │
│    ├─ 挂载网络 (comp_network)               │
│    └─ 创建容器                              │
│                                            │
│  start_container() / stop_container()       │
│  remove_container()                         │
│  get_container_status() / get_container_logs() │
│  exec_in_container() — 用于判题             │
└────────────────────────────────────────────┘
```

**端口分配算法：**
1. 扫描所有现有容器，收集已占用 HostPort
2. 在 [30000, 40000) 区间内返回第一个未占用的端口
3. 若扫描失败，随机返回区间内端口

**资源隔离策略：**
- CPU: `--nano-cpus` 限制（如 0.5 核 = 500000000 nano CPUs）
- 内存: `--mem_limit` + `--memswap_limit` 双重限制（防止 swap 逃逸）
- 网络: 独立 bridge 网络 `comp_network`，容器间网络隔离

### 4.3 环境部署服务 (environment_service.py)

```
deploy_competition_environments(comp_id)
  │
  ├─ 获取竞赛信息 + 资源限制参数
  ├─ 获取所有选手 + Docker 题目 (跳过试卷)
  │
  ├─ for 选手 × 题目:  (双重循环)
  │    ├─ 检查是否已有运行中的容器 → 跳过
  │    ├─ 构建容器名: comp-{comp_id}-u{user_id}-c{chal_id}
  │    ├─ 提取 EXPOSE 端口
  │    ├─ 检查镜像是否已构建 → 未构建则标记失败
  │    ├─ create_container() → 创建容器
  │    └─ 写入 Environment 记录
  │
  ├─ 记录部署指标 (总耗时、成功/失败数、平均耗时)
  └─ 通过 SocketIO 实时推送进度
```

### 4.4 判题引擎 (judge_service.py)

```
判题方式:
  ├─ port 检测: ss/netstat 检查容器内端口监听
  ├─ command 检测: docker exec 执行命令，比对输出
  ├─ file 检测: test -f 检查文件是否存在
  └─ exam 判题: 逐题比对答案，60% 正确率通过

自动调度:
  ├─ 每 30s: 自动判题 active 竞赛的所有环境
  └─ 每 120s: 自动判题+停止+清理 finished 竞赛
```

### 4.5 自动构建调度器

```
┌──────────────────────────────────────────────────────┐
│                 APScheduler 定时任务                    │
│                                                      │
│  auto_build_job (每 15s)                              │
│    └─ 查找 active 且 auto_deployed=false 的竞赛       │
│       └─ deploy_competition_environments()            │
│          └─ 标记 auto_deployed=true                   │
│                                                      │
│  auto_judge_job (每 30s)                              │
│    └─ 遍历 active 竞赛的所有 Docker 环境              │
│       └─ judge_environment() 逐个判题                │
│                                                      │
│  auto_cleanup_job (每 120s)                           │
│    └─ 遍历 finished 竞赛                              │
│       ├─ 最后一批判题                                  │
│       ├─ 停止所有容器                                  │
│       └─ 删除所有容器 + 环境记录                       │
└──────────────────────────────────────────────────────┘
```

### 4.6 试卷考试子系统 (扩展功能)

系统除 Docker 实操题外，还支持传统试卷考试模式，作为平台的扩展功能。

```
┌──────────────────────────────────────────────────────┐
│                   试卷考试流程                          │
│                                                      │
│  [管理员组卷]                                         │
│    ├─ 创建题目 → 选择类型「试卷考试」                    │
│    ├─ 添加小题: 单选题 / 判断题 / 填空题                │
│    │   ├─ 单选题: options=["A.xx","B.xx"], answer="A"  │
│    │   ├─ 判断题: options=["正确","错误"], answer="正确" │
│    │   └─ 填空题: options=null, answer="nginx"         │
│    └─ 逐题设分                                        │
│                                                      │
│  [选手答题]                                           │
│    ├─ 进入试卷页 → 逐题作答                            │
│    ├─ 单选/判断: radio 选择                           │
│    ├─ 填空: text input                               │
│    └─ 提交 → AJAX → 服务端判题                        │
│                                                      │
│  [自动判分]                                           │
│    ├─ 逐题比对答案 (填空不区分大小写)                    │
│    ├─ 计算总分                                        │
│    ├─ 60% 正确率 → 通过                               │
│    └─ 写入 Score 表 (answers JSON 含逐题详情)          │
└──────────────────────────────────────────────────────┘
```

**数据模型：** 试卷题目通过 `Challenge.challenge_type = "exam"` 标识，小题存储在独立的 `ExamQuestion` 表中，支持 `single_choice`、`true_false`、`fill_blank` 三种题型，选项以 JSON 格式存储。

**与 Docker 题的并行关系：** 同一竞赛可同时包含 Docker 实操题和试卷题，部署时自动跳过试卷题（不创建容器），判题时分别处理。排行榜汇总两种题型的得分。

## 6. 自动化构建流水线 (核心论文素材)

### 6.1 完整自动化流程

```
[管理员创建竞赛] → [添加 Docker 题目] → [编写 Dockerfile / 选模板]
                                              │
                                              ▼
                                     [构建镜像] (带重试)
                                              │
                    ┌─────────────────────────┤
                    │                         │
              [管理员点"开始"]           [到达 start_time]
                    │                         │
                    ▼                         ▼
          ┌─────────────────────────────────────────┐
          │        自动构建调度器触发                  │
          │  deploy_competition_environments()       │
          │                                         │
          │  for 选手:                               │
          │    for 题目:                             │
          │      ① 分配端口 (30000-40000 扫描)        │
          │      ② 设置资源限制 (CPU / Memory)        │
          │      ③ 创建容器 (bridge 网络隔离)          │
          │      ④ 写入 Environment 记录              │
          │      ⑤ SocketIO 推送进度                  │
          │                                         │
          │  标记 auto_deployed = true               │
          │  记录部署指标 (耗时、成功率)                │
          └─────────────────────────────────────────┘
                    │
                    ▼
          [选手操作容器完成题目]
                    │
                    ▼
          ┌─────────────────┐
          │  自动判题 (30s)   │
          │  - 端口检测       │
          │  - 命令执行       │
          │  - 文件检测       │
          └─────────────────┘
                    │
                    ▼
          [竞赛结束 → 自动清理]
          ┌─────────────────┐
          │  自动清理 (120s)  │
          │  ① 最后一轮判题   │
          │  ② 停止所有容器   │
          │  ③ 删除所有容器   │
          └─────────────────┘
```

### 6.2 自动化程度对照

| 环节 | 传统方式 | 本系统 |
|------|---------|--------|
| 环境搭建 | 手动为每个选手配置 VM | **自动为 N 个选手 × M 个题目创建 M×N 个容器** |
| 端口分配 | 手动指定/冲突排查 | **自动扫描空闲端口 (30000-40000)** |
| 资源限制 | 依赖人工配置 | **CPU/Memory 自动限制，防止资源争抢** |
| 判题评分 | 人工检查 | **每 30s 自动判题，结果写入数据库** |
| 环境回收 | 手动清理 | **竞赛结束后自动停止+删除容器** |
| 构建失败 | 手动重试 | **自动重试 2 次，带日志** |

## 9. 关键性能指标

系统应记录以下指标供论文使用：

- **并发部署能力**: 同时为 P 个选手 × C 个题目创建 P×C 个容器
- **平均部署耗时**: 单容器创建时间 + 端口分配开销
- **端口利用率**: [30000, 40000] 区间可支持最多 10000 个并发容器
- **判题延迟**: 单次判题从触发到返回结果的时间
- **资源开销**: 每个容器的 CPU/Memory 占用基线
